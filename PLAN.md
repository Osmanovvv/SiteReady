# SiteReady — полный план проекта (от А до Я)

Веб-приложение для проверки сайта «под ключ»: за один прогон обходит весь сайт и
выдаёт единый отчёт по SEO, техническому QA, производительности, доступности и
адаптивности — с оценкой, приоритетами и советами по исправлению.

**Реализация:** фронтенд — в **Lovable** (React + Vite + Tailwind + shadcn), движок и бэкенд —
**Node.js** (пилится через **Claude Code**). Между ними единый контракт данных: `CONTRACT.md`
(схема отчёта + SSE), мок для фронта — `sample-report.json`, ТЗ для Lovable — `lovable-prompt.md`.

---

## 1. Цель и определение «100% готово»

После выполнения плана:

- запускается одной командой `node server.js` (Node отдаёт собранный фронт Lovable + API), открывается интерфейс в браузере;
- вводишь URL → инструмент обходит весь сайт и показывает прогресс в реальном времени;
- считаются все 5 категорий проверок, общий балл и буквенная оценка (A–F);
- проблемы сгруппированы по категориям и приоритету, с понятным советом «как починить»;
- есть постраничная таблица результатов;
- два режима отчёта: «Быстрый вид» (для себя) и «Клиентский отчёт» (для сдачи);
- экспорт клиентского отчёта в PDF (печать) и в автономный HTML-файл;
- проходит автотест на локальном тестовом сайте с заранее заложенными ошибками;
- есть README; процесс устойчив к таймаутам, редиректам и не-HTML страницам.

## 2. Технические решения

- **Движок и бэкенд — чистый Node.js, ноль внешних зависимостей.** Только встроенные модули:
  `http`, `https`, `zlib`, `url`, `fs`, `path`. Нечему ломаться, нет supply-chain.
  (Относится к движку; фронт — отдельное React-приложение из Lovable, см. ниже.)
- **CommonJS** (`require`) в движке — максимально предсказуемо.
- **Фронтенд — в Lovable** (React + Vite + Tailwind + shadcn): отдельное приложение,
  общается с движком строго по `CONTRACT.md`. Прогресс обхода — через **SSE**. Финально
  Node отдаёт собранный бандл фронта как статику + сам API → запуск одной командой.
- **Статический анализ** (без headless-браузера) — покрывает ~80% реальных проблем
  при сдаче и работает быстро и стабильно. Проверки, которым нужен реальный рендер
  (ошибки в JS-консоли, точный контраст, скриншоты), вынесены в «Фазу 2» как
  опциональный модуль на Puppeteer (см. §14).
- **Честно про цену этого размена.** Заметная доля современных сайтов (React/Vue, тяжёлый
  Bitrix, часть Tilda) рендерит контент на клиенте — для них v1 ограничен: покажем баннер
  и переклассифицируем находки (см. §5.4), но полноценный контент увидим только в Фазе 2.
  Это сознательный выбор «ноль зависимостей ↔ охват» в пользу надёжности v1, и это **потолок v1**.

## 4. Каталог проверок (что именно проверяем)

Серьёзность: **critical** (красное, чинить обязательно) · **warning** (жёлтое, желательно) · **info** (синее, на заметку).

### 4.1 SEO
- Title: наличие (critical), длина 30–60 символов (warning), уникальность по сайту (warning), не плейсхолдер вроде «Document» (info).
- Meta description: наличие (warning), длина 70–160 (info), уникальность (warning).
- Ровно один `<h1>` (0 — critical, >1 — warning).
- Порядок/пропуски уровней заголовков — проверяются и штрафуются в Доступности (§4.4), здесь не дублируются (чтобы не было двойного счёта).
- `<link rel="canonical">` присутствует (warning).
- `meta robots noindex/nofollow` — подсветить (warning: страница может не индексироваться).
- Open Graph: `og:title`, `og:description`, `og:image` (warning — иначе кривые превью в соцсетях/мессенджерах).
- Twitter Card (info).
- Микроразметка JSON-LD: наличие (info), валидный JSON (warning при битом), наличие `@type` и обязательных полей для типа (Article→`headline`, Product/Offer→`price` и т.п.) — warning при неполноте, иначе rich result не покажется.
- Атрибут `lang` у `<html>` (warning).
- Favicon (info).
- Уровень сайта: `robots.txt` существует (warning), `sitemap.xml` существует и указан в robots (warning).

### 4.2 Технические / QA
- Битые внутренние ссылки 4xx/5xx (critical).
- Битые внешние ссылки (warning). Различаем настоящую битость (`404/410`, отказ соединения, DNS) и «не удалось проверить» (`403/405/429/таймаут` — боту просто не ответили): второе не штрафуем. Запросы с реалистичным `User-Agent`. См. §5.4.
- Битые картинки (src не 200 или не картинка) (critical).
- Mixed content: http-ресурс на https-странице (critical).
- Сайт отдаётся по HTTPS (critical, если нет).
- Редирект http→https настроен (warning, если нет).
- Цепочки редиректов длиннее 1 перехода (info/warning).
- «Мягкий 404»: несуществующий путь отдаёт 200 вместо 404 (warning).
- Страницы с не-200 статусом, найденные при обходе (critical).
- `target="_blank"` без `rel="noopener"` (warning — безопасность).
- Пустые ссылки `href="#"`/пустой href (info).
- Объявление кодировки: есть `meta charset` или `charset` в `Content-Type` (warning, если нигде — частый дефект legacy/Bitrix, ломает рендеринг). Дополняет детект кодировки из §5.4.
- Битые якоря: ссылка `#section` ведёт на существующий `id` на странице (warning, если `id` нет — мёртвые оглавления/навигация).

### 4.3 Производительность (измеряем реально)
- TTFB: хорошо <200 мс, warning 200–600, плохо >600. **Информативно — в балл не идёт (сетезависимо, см. §6).**
- Размер HTML-документа: warning >100 КБ, плохо >300 КБ.
- Суммарный вес страницы (HTML + ресурсы) — **оценочно по `Content-Length` из HEAD, без полной загрузки**: warning >2 МБ, плохо >4 МБ. Идёт в балл; при отсутствии `Content-Length` помечается как неполный.
- Сжатие (gzip/br) для HTML/CSS/JS (warning, если текст без сжатия).
- Заголовки кэширования (`Cache-Control`/`Expires`) на статике (info/warning).
- Число запросов на страницу (info >50, warning >100).
- Тяжёлые картинки >200 КБ (warning, с примерами).
- Картинки без `width`/`height` — риск скачков верстки CLS (warning).
- Картинки без `loading="lazy"` (info).
- Рендер-блокирующие скрипты в `<head>` без `async`/`defer` (warning); число CSS-файлов (info).
- Большой объём инлайн CSS/JS (info).

> TTFB и общее время загрузки — сетезависимы (меряются с точки запуска), показываются как «оценочно» и **в балл не входят** (см. §6). Остальные сигналы — сжатие, размеры/вес, кэш, рендер-блок, число запросов — воспроизводимы и идут в балл.

### 4.4 Доступность (a11y, статическая часть)
- Картинки без `alt` (critical, со счётчиком и примерами).
- Нет `lang` у `<html>` (warning).
- Поля форм без `<label>`/`aria-label` (warning).
- Кнопки/ссылки без текста и без `aria-label` (иконки) (warning).
- `viewport` с `user-scalable=no`/`maximum-scale=1` — блокирует зум (warning).
- Generic-тексты ссылок «тут», «подробнее» (info).
- Порядок заголовков без пропусков уровней (info — владелец этой проверки здесь, см. §4.1).
- Дубли `id` на странице (warning — WCAG 4.1.1; ломает `label for`, якоря, ARIA).
- `tabindex` > 0 (warning — ломает естественный порядок фокуса).
- Таблицы с данными без `<th>`/`scope` (warning).
- `autocomplete` на полях идентичности (email/имя/телефон) (info).
- Контраст цветов и порядок фокуса с клавиатуры — deep-проверки Фазы 2 (нужен рендер).

### 4.5 Адаптивность
- `meta viewport` присутствует (critical, если нет).
- `viewport` со `width=device-width` (warning, если иначе).
- Медиа-запросы `@media` найдены в CSS (warning, если их нет — вероятно не адаптив).
- Фиксированные большие ширины в инлайн-стилях / широкие таблицы и картинки (info).
- Риск горизонтального скролла: элементы шире вьюпорта (info, эвристика).

## 5. Движок

### 5.1 Fetcher (`src/fetcher.js`)
Запрос через встроенные `http`/`https` с реалистичным `User-Agent`: ручное следование
редиректам с записью цепочки; замер TTFB и общего времени; захват статуса, типа контента
и заголовков (`content-encoding`, `cache-control`, `content-length`); декомпрессия
gzip/deflate/br через `zlib` с **ограничением размера распакованного тела** (защита от
gzip-бомб). Тело страниц/CSS возвращается **сырым `Buffer`** — превращаем в строку только
после определения кодировки (см. §5.4). Для страниц/CSS — GET; для прочих ресурсов — HEAD
(с откатом на GET). Жёсткие таймауты. Перед каждым запросом, включая каждый редирект-хоп,
прогоняем **egress-guard по разрешённому IP** (§7).

### 5.2 Crawler (`src/crawler.js`)
BFS от стартового URL, только тот же сайт (host и его `www`-вариант). Нормализация
ссылок (абсолютизация, учёт `<base href>`, отбрасывание `#`-якорей, дедупликация,
пропуск `mailto:`/`tel:`/`javascript:`). Лимит страниц (по умолчанию 50, в UI — больше),
параллелизм (5), колбэк прогресса на каждую страницу. Сбор всех ссылок и ресурсов
(img/script/link/iframe) для последующих проверок.

### 5.3 Парсер (`src/html.js`)
**Не регэксп, а маленький прощающий токенайзер** (конечный автомат): корректно
обрабатывает комментарии `<!-- -->`, `<!DOCTYPE>`, raw-text элементы (`script`, `style`,
`textarea`, `title`) — где `<` не является тегом, — атрибуты в двойных/одинарных/без
кавычек/пустые, самозакрытие и битую разметку без падения. Извлекает только нужное:
title, meta, link, h1–h6 (в порядке документа), ссылки, картинки, скрипты (вкл. инлайн
JSON-LD), iframe/source, инлайн `<style>` и подключённый CSS. Это снимает главный риск —
ложные срабатывания на грязном HTML (для аудита ложная находка хуже пропущенной).
Покрывается фикстурой с намеренно кривым HTML.

### 5.4 Достоверность результатов (анти-ложные-срабатывания)
- **Кодировки cp1251/koi8-r и др.** Много российских сайтов (Bitrix, старые проекты) в
  `windows-1251`. Определяем кодировку: `Content-Type` → `<meta charset>` **и старая форма
  `<meta http-equiv="Content-Type" content="…charset=windows-1251">`** (снифф первых ~2 КБ
  как latin1) → дефолт utf-8; декодируем встроенным `new TextDecoder(label)` (ICU есть в
  Node по умолчанию). Без этого — кракозябры и ложные находки по всем категориям.
- **Детект SPA/клиентского рендеринга.** Если контент рисуется в браузере (мало видимого
  текста + корневой `<div id=root/app>`/маркеры React/Vue/Angular + крупный JS-бандл) —
  ставим флаг, показываем честный баннер «статический аудит ограничен, нужен deep-режим
  Фазы 2», а находки «нет h1/нет alt/тонкий контент» **переклассифицируем** из critical в
  предупреждение с пояснением (а не сыпем десятком ложных critical).
- **Классификация ссылок.** Реалистичный `User-Agent`. `404/410`, отказ соединения и
  DNS-ошибка → действительно битая (critical для внутренних). `403/405/429/503/таймаут/999`
  → «не удалось проверить» (нейтрально, без штрафа) — иначе раздел Tech/QA фонит.
- **Производительность — оценочно.** TTFB и вес меряются с машины аудита (без CDN-гео,
  HTTP/2 и браузерного параллелизма). В отчёте — формулировка «технические сигналы,
  оценочно»; акцент на сетенезависимом: сжатие, вес картинок, кэш-заголовки, рендер-блок.

## 6. Модель оценки (скоринг) — `src/score.js`

**Балл категории = 100 − Σ(штрафы проверок), обрезка [0,100].** Каждая проверка штрафует
**ровно одну** категорию (таблица владения ниже) — двойного счёта нет.

**Базовый потолок штрафа по серьёзности** (максимум, что одна проверка снимает с категории):
critical — 30, warning — 12, info — 4.

**Два режима штрафа (оба ограничены потолком):**
- **prevalence** (качество страниц: нет title, нет alt, нет viewport…):
  `штраф = потолок × (страниц_с_проблемой / просканировано)`. На всех страницах → полный
  потолок; на половине → половина; на 1 из 300 — почти ноль (это и есть «для сайта в целом незначительно»).
- **count** (дискретные дефекты: битые ссылки, mixed-content, битые картинки):
  `штраф = min(потолок, цена_за_шт × количество)`, цена_за_шт = потолок/8 (≈8 шт насыщают потолок).
  Так 200 `<img>` без alt не уводят в −∞ (потолок 30), а 1 битая ссылка на 300-страничном
  сайте всё равно весит — в отличие от prevalence. **Это прямой ответ на «не должно быть −∞».**

**Не входит в балл (только info):** сетезависимые метрики перфа — **TTFB и общее время
загрузки** — выведены из оценки, показываются как «оценочно, с точки замера». В балл перфа
идут только воспроизводимые сигналы: сжатие, вес картинок, суммарный вес (оценочно по
`Content-Length`), `width/height`, рендер-блок, кэш-заголовки, число запросов, размер HTML.

**Владение проверками (каждый дефект — одна категория):**
- **SEO** — title, meta description, один H1, canonical, noindex, OG, JSON-LD, favicon, robots.txt, sitemap.xml.
- **Доступность (статич.)** — alt, `html lang`, label форм, пустые/иконочные контролы, блокировка зума (`user-scalable=no`), порядок заголовков, generic-тексты ссылок, дубли `id`, `tabindex>0`, таблицы без `<th>`/scope, `autocomplete` на полях идентичности.
- **Адаптивность** — наличие `viewport`, `width=device-width`, наличие `@media`, риск overflow.
- **Производительность** — сжатие, вес картинок, суммарный вес (оценочно из `Content-Length`), `width/height`, рендер-блок, кэш, число запросов, размер HTML.
- **Тех/QA** — битые ссылки/картинки, mixed content, HTTPS, редирект http→https, цепочки редиректов, мягкий 404, `target=_blank` без noopener, не-200 страницы, объявление charset, битые якоря (`#id`).

Пересечения разведены по разным под-проверкам: `lang` штрафует только a11y (в SEO — info-ссылка
без штрафа); `alt` — только a11y; наличие `viewport` — только адаптив, а блокировка зума — только
a11y (разные дефекты одного тега).

**Веса категорий в общем балле:** SEO 0.30 · Тех/QA 0.30 · Производительность 0.18 ·
Доступность (статич.) 0.12 · Адаптивность 0.10 (сумма 1.00). Сетезависимое из веса убрано,
поэтому общий балл стоит на воспроизводимых сигналах.

**Оценка:** A ≥90, B ≥75, C ≥60, D ≥40, F <40.

**Достоверность и охват (показываем рядом с баллом):**
- бейджи у категорий: a11y — «проверено статически; контраст/клавиатура — в deep-режиме»;
  перф — «скоростные метрики оценочны»;
- **охват** — «проверено N из ~M найденных страниц» + бейдж «выборка», если сработал лимит.
  Численно за выборку балл не штрафуем (произвольно), но честно помечаем: это оценка по
  выборке, а не вердикт по всему сайту.

**Калибровка (обязательна):** структура формулы зафиксирована выше; на фикстурах
подбираются только числа (потолки/цены/веса) так, чтобы «очевидно плохой» сайт давал D/F,
«хороший» — A/B; есть тест на граничное значение (страница ровно на пороге A/B).

## 7. Backend (API) — `server.js`
- **Ingress (кто достучится до сервера):** слушаем только `127.0.0.1` — сервер не виден из чужой сети. Это про входящие; к SSRF отношения не имеет.
- **Egress / SSRF (куда ходит фетчер):** отдельный guard в `fetcher.js`. Фетчер по сабмиту пользователя дёргает произвольный URL и идёт по редиректам, поэтому валидируем **назначение**: резолвим DNS один раз и **коннектимся ровно по проверенному IP** через кастомный `lookup`-хук `http(s)`-запроса (без повторного резолва на коннекте → нет TOCTOU-окна — это и есть честный анти-rebinding), на стартовом URL, **каждом хопе редиректа** и каждой внешней проверке; блокируем loopback / link-local / RFC1918 / ULA `fc00::/7` / CGNAT `100.64/10` / `0.0.0.0`. Метадата-IP `169.254.169.254` блокируем **всегда**. Снятие — только явным `allowPrivate` (для аудита своего `localhost` в деве); при блокировке — понятное сообщение, а не молчаливый таймаут.
- `GET /` и статика → **собранный фронт Lovable** (`web/dist`).
- `GET /api/audit/stream?url=&maxPages=&external=&allowPrivate=` → **SSE** (`meta` /
  `progress` / `done` / `error`). **Точная схема событий и отчёта — в `CONTRACT.md`.**
  Обрыв соединения клиентом останавливает обход.
- Состояние в памяти, без БД. Один прогон на запрос.

## 8. Frontend (UX) — Lovable (`web/`)
- Собирается в **Lovable** по `lovable-prompt.md`; данные — строго по `CONTRACT.md`, мок — `sample-report.json`. Требования к UI:
- **Ввод**: поле URL, кнопка «Проверить», доп. опции (лимит страниц, проверять внешние ссылки, разрешить локальные адреса).

## 10. Тестирование
Локальный сайт с намеренными дефектами (нет alt, нет meta description, дублирующиеся
title, битая ссылка, отсутствует viewport, нет h1, тяжёлый инлайн и т.д.), **плюс
отдельные фикстуры: эталонно-чистая страница (валидный HTML, все мета на месте — верхний
конец шкалы и контроль ложных срабатываний), страница в windows-1251, страница с кривым
HTML, заглушка SPA**. `test/run.js` запускает движок против фикстур и проверяет, что
каждая заложенная проблема найдена, кодировка читается, SPA детектится, парсер не фонит,
а на чистой странице находок почти нет, а баллы **откалиброваны** (плохой сайт → D/F,
хороший → A/B). **Отдельные фикстуры на устойчивость:** цепочка и цикл редиректов,
медленный эндпоинт (таймаут), ответ сверх лимита распаковки. **Граничный тест калибровки:**
страница, собранная ровно на порог A/B. Дополнительно: сервер отдаёт UI, SSE отдаёт
корректный JSON (через curl к localhost). Обязательный шаг приёмки.

## 11. Краевые случаи и устойчивость
Таймауты; циклы редиректов (visited-set + лимит хопов, §12); не-HTML стартовый URL;
gzip/br-декод с лимитом размера (анти gzip-бомба); **кодировки utf-8/windows-1251/koi8-r
через `TextDecoder` с детектом по Content-Type и `meta charset`** (см. §5.4); **детект SPA**
и переклассификация находок; относительные и протокол-относительные URL; `<base href>`;
дедуп по якорю/слэшу; пропуск `mailto/tel/js`; ограничение размера одного тела **и общий
байт-бюджет всего обхода** (страниц×байт, не только одно тело); бинарные ресурсы только
через HEAD; строгий host + объединение `www`. **Egress-guard против SSRF на каждое
назначение — см. §7.** **Вежливость обхода:** реалистичный `User-Agent`, троттлинг на хост
+ джиттер, ограниченный параллелизм и лимит внешних проверок — чтобы не выглядеть мини-DoS
и не ловить rate-limit. `robots.txt` читается и показывается; подчинение Disallow — опция
(по умолчанию выкл., т.к. аудитишь свой сайт).

## 12. Дефолты и настройки
maxPages 50 (UI до ~300); параллелизм 5 с троттлингом ~150 мс на хост (+джиттер);
реалистичный `User-Agent`; таймаут запроса 15 c; **лимит редирект-хопов 5** (+visited-set
против циклов); проверка внешних ссылок вкл., лимит 150 уникальных, таймаут 8 c, HEAD→GET;
лимит распаковки одного тела ~8 МБ, **общий байт-бюджет обхода ~150 МБ**; `allowPrivate`
выкл. по умолчанию (SSRF-guard, §7); пороги — из §4.

## 14. Ограничения и развитие (v2)
Декодирование не-utf8 (cp1251/koi8-r) опирается на встроенный `TextDecoder`, т.е. на
полный ICU — он есть в стандартной сборке Node ≥13; на редкой сборке `small-icu` не-utf8
страницы деградируют (отмечено в README). Опциональный deep-модуль на Puppeteer: реальные
ошибки JS-консоли, контраст и полный a11y через axe-core, скриншоты на брейкпоинтах,
полноценный Lighthouse. Также: сайты под
авторизацией, расписание + история аудитов с диффом, мультисайтовый дашборд, white-label
брендирование клиентских отчётов, серверный PDF, интернационализация.
