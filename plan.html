PLAN.md — SiteReady

SiteReady — полный план проекта

Веб-приложение для проверки сайта «под ключ»: за один прогон обходит весь сайт и выдаёт единый отчёт по SEO, техническому QA, производительности, доступности и адаптивности — с оценкой, приоритетами и советами по исправлению.

1 · Цель и определение «100% готово»

  • запуск одной командой node server.js, интерфейс в браузере;
  • URL → обход всего сайта с прогрессом в реальном времени;
  • 5 категорий проверок, общий балл и буквенная оценка A–F;
  • проблемы сгруппированы по категориям и приоритету, с советом «как починить»;
  • постраничная таблица результатов;
  • два режима: «Быстрый вид» и «Клиентский отчёт»;
  • экспорт отчёта в PDF (печать) и автономный HTML;
  • автотест на локальном сайте с заложенными ошибками; README; устойчивость к таймаутам и редиректам.

2 · Технические решения

  • Чистый Node.js, ноль зависимостей — только http, https, zlib, url, fs, path. Нет npm install, нечему ломаться у клиента.
  • CommonJS — максимально предсказуемо.
  • Лёгкий HTTP-сервер + статика; прогресс обхода через SSE.
  • Статический анализ покрывает ~80%; рендер-зависимое (JS-консоль, контраст, скриншоты) — в «Фазу 2» на Puppeteer.

3 · Архитектура и структура файлов

site-ready/ package.json без dependencies; start / cli / test server.js HTTP-сервер: статика + SSE-аудит cli.js запуск аудита из терминала README.md / PLAN.md src/ fetcher.js запрос: тайминги, редиректы, gzip/br html.js прощающий токенайзер HTML crawler.js BFS-обход внутренних ссылок audit.js оркестратор: обход → проверки → скоринг score.js веса, расчёт баллов и оценки checks/ seo · tech · performance · accessibility · responsive public/ index.html · app.js · styles.css test/ fixtures/ · serve-fixtures.js · run.js

4 · Каталог проверок

Серьёзность: critical чинить обязательно · warning желательно · info на заметку.

4.1 SEO

  • Title: наличие crit, длина 30–60 warn, уникальность warn, не плейсхолдер info.
  • Meta description: наличие warn, длина 70–160 info, уникальность warn.
  • Ровно один <h1> (0 — crit, >1 — warn); иерархия заголовков info.
  • canonical warn; robots noindex warn; Open Graph warn; Twitter Card info.
  • JSON-LD: наличие info, валидность warn; lang у html warn; favicon info.
  • Сайт-уровень: robots.txt warn, sitemap.xml + ссылка в robots warn.

4.2 Технические / QA

  • Битые внутренние ссылки 4xx/5xx crit; битые картинки crit; не-200 при обходе crit.
  • Битые внешние ссылки warn — различаем реальную битость (404/410, отказ, DNS) и «не удалось проверить» (403/405/429/таймаут, не штрафуем). Реалистичный User-Agent.
  • Mixed content crit; HTTPS crit; редирект http→https warn; цепочки редиректов info.
  • Мягкий 404 warn; target=_blank без rel=noopener warn; пустые href info.

4.3 Производительность

  • TTFB <200 / 200–600 / >600 мс; вес HTML >100/300 КБ; суммарный вес >2/4 МБ.
  • Сжатие gzip/br warn; кэш-заголовки info; число запросов >50/100; тяжёлые картинки >200 КБ warn.
  • Картинки без width/height (CLS) warn, без loading=lazy info; рендер-блок скрипты в <head> warn.

Метрики скорости — оценочные, с точки замера. Сетенезависимые сигналы (сжатие, вес картинок, кэш, рендер-блок) — достоверны.

4.4 Доступность (a11y, статика)

  • Картинки без alt crit; нет lang warn; поля форм без label warn.
  • Иконки-кнопки без aria-label warn; user-scalable=no warn; generic-тексты ссылок info.
  • Контраст — deep-проверка Фазы 2 (нужен рендер).

4.5 Адаптивность

  • meta viewport crit; width=device-width warn; @media в CSS warn.
  • Фиксированные большие ширины / широкие таблицы info; риск гор. скролла info (эвристика).

5 · Движок

5.1 Fetcher

http/https с реалистичным User-Agent: ручные редиректы с цепочкой; TTFB и общее время; декомпрессия gzip/deflate/br через zlib с лимитом размера распакованного тела (анти gzip-бомба). Тело — сырой Buffer, в строку только после определения кодировки. Страницы/CSS — GET; прочее — HEAD→GET.

5.2 Crawler

BFS, только тот же сайт (host + www). Нормализация ссылок (<base>, без #, дедуп, пропуск mailto/tel/js). Лимит страниц 50 (UI больше), параллелизм 5, колбэк прогресса.

5.3 Парсер

Не регэксп, а маленький прощающий токенайзер (конечный автомат): комментарии, DOCTYPE, raw-text (script/style/textarea/title), атрибуты в любых кавычках и без, самозакрытие, битая разметка без падения. Снимает главный риск — ложные срабатывания на грязном HTML.

5.4 Достоверность результатов

  • Кодировки cp1251/koi8-r. Определяем: Content-Type → <meta charset> и старая форма http-equiv Content-Typeобновлено → utf-8; декод через TextDecoder (ICU в Node).
  • Детект SPA. Контент рисуется в браузере → флаг + честный баннер, а находки «нет h1 / нет alt / тонкий контент» переклассифицируем, а не сыпем ложными critical.
  • Классификация ссылок. 404/410/отказ/DNS → битая; 403/405/429/таймаут → «не проверено», без штрафа.
  • Производительность — оценочно. Акцент на сетенезависимом.

6 · Скоринг

  • Штраф за находку с потолком на тип. Веса: critical 15, warning 6, info 2.
  • Балл категории = clamp(0, 100 − штраф).
  • Общий = взвешенное среднее: SEO 0.30 · Тех 0.30 · Перф 0.20 · a11y 0.12 · Адаптив 0.08.
  • Оценка: A ≥90 · B ≥75 · C ≥60 · D ≥40 · F <40.

7 · Backend — server.js

  • Слушает только 127.0.0.1 — закрываем SSRF, т.к. бэкенд ходит по произвольным URL.
  • GET / → UI; GET /api/audit/stream → SSE: события meta · progress · done · error. Обрыв клиента останавливает обход.
  • Состояние в памяти, без БД.

8 · Frontend — public/

  • Ввод: URL, опции (лимит страниц, внешние ссылки, deep — выкл. в v1).
  • Прогресс: полоса, «X из Y», текущий URL, фазы.
  • Дашборд: циферблат балла + оценка, 5 карточек категорий, список проблем с фильтром и советом, постраничная таблица.
  • Переключатель «Быстрый вид / Клиентский отчёт»; кнопки «Скачать PDF / HTML». Весь UI на русском, адаптивный.

10 · Тестирование

Локальный сайт с дефектами + фикстуры: эталонно-чистая страницаобновлено, windows-1251, кривой HTML, заглушка SPA. test/run.js проверяет: каждая ошибка найдена, кодировка читается, SPA детектится, парсер не фонит, на чистой странице находок почти нет, баллы откалиброваны (плохой → D/F, хороший → A/B).

11 · Краевые случаи

Таймауты; циклы редиректов; не-HTML старт; gzip/br с лимитом; кодировки utf-8/1251/koi8-r через TextDecoder; детект SPA; относительные и протокол-относительные URL; <base>; дедуп; вежливость обхода (User-Agent, троттлинг на хост + джиттер, лимиты); robots.txt читается, Disallow — опция (по умолч. выкл.).

12 · Дефолты

maxPages 50 (UI ~300); параллелизм 5 + троттлинг ~150 мс/хост (+джиттер); таймаут 15 c; внешние ссылки лимит 150, таймаут 8 c, HEAD→GET; лимит распаковки ~8 МБ.

13 · Поставляемые файлы

Полный исходник (server.js, src/*, public/*, cli.js), package.json, README.md (RU), PLAN.md, тестовый набор. Готово к node server.js.

14 · Ограничения и развитие (v2)

Не-utf8 декод опирается на полный ICU (стандартный Node ≥13); на small-icu деградируетобновлено. Deep-модуль на Puppeteer: JS-консоль, контраст через axe-core, скриншоты, Lighthouse. Также: авторизация, история с диффом, мультисайтовый дашборд, white-label, серверный PDF, i18n.

15 · Критерии приёмки

  • node server.js стартует, UI открывается
  • URL → обход с живым прогрессом; 5 категорий, балл, оценка
  • проблемы сгруппированы; постраничная таблица; 2 режима; экспорт PDF/HTML
  • автотест на фикстурах проходит; README точный; краевые случаи не роняют процесс
  • cp1251 без кракозябр; SPA даёт баннер; парсер не фонит на кривом HTML
  • внешние 403/405/429 не «битые»; калибровка: плохой → D/F, хороший → A/B
