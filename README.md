# SiteReady — аудит сайта «под ключ»

[![CI](https://github.com/Osmanovvv/SiteReady/actions/workflows/ci.yml/badge.svg)](https://github.com/Osmanovvv/SiteReady/actions/workflows/ci.yml)
![Node](https://img.shields.io/badge/node-%E2%89%A518-339933?logo=node.js&logoColor=white)
![deps](https://img.shields.io/badge/engine-zero%20dependencies-success)
![tests](https://img.shields.io/badge/tests-106%20engine%20%2B%206%20web-success)
![license](https://img.shields.io/badge/license-MIT-blue)

Веб-приложение, которое за один прогон обходит сайт и выдаёт единый отчёт по
**SEO · техническому QA · производительности · доступности · адаптивности** —
с общим баллом, буквенной оценкой A–F, приоритизированными проблемами и советами
«как починить». Есть «Быстрый вид» для себя и «Клиентский отчёт» для сдачи с
экспортом в PDF и автономный HTML.

| | |
|---|---|
| **Движок и бэкенд** | чистый Node.js, **ноль внешних зависимостей** (только встроенные `http`/`https`/`zlib`/`dns`/…) |
| **Фронтенд** | React + Vite SPA (TanStack Router, Tailwind, shadcn/ui), собирается в статику |
| **Запуск** | одна команда — `node server.js` |
| **Требования** | Node.js ≥ 18 (рекомендуется 20+) |

## Быстрый старт

Фронт уже собран в `web/dist`, поэтому после клонирования достаточно одной команды:

```bash
git clone https://github.com/Osmanovvv/SiteReady.git
cd SiteReady
node server.js
```

Откройте **http://127.0.0.1:3000** — введите адрес сайта и нажмите «Проверить».
Порт можно сменить через переменную окружения `PORT`.

> Сервер слушает **только `127.0.0.1`** — он не виден из чужой сети.

### Пересборка фронта (если меняли исходники)

```bash
cd web
npm install
npm run build      # → web/dist (его отдаёт server.js)
```

## Командная строка (без UI)

```bash
node cli.js https://example.com                 # полный JSON-отчёт
node cli.js https://example.com --summary       # краткая сводка в терминал
node cli.js https://example.com --max=100       # лимит страниц
node cli.js https://example.com --external      # + проверка внешних ссылок
node cli.js http://127.0.0.1:8080 --allow-private   # аудит локального адреса
node cli.js https://spa-site.ru --deep              # глубокий режим (браузер)
```

## Возможности v2

- **Глубокий режим** — рендер каждой страницы в реальном браузере (Playwright/Chromium):
  закрывает слепую зону **SPA** (React/Vue), меряет **реальный контраст** (axe-core),
  ловит **ошибки JS-консоли**. Точнее, но медленнее; лимит страниц ≤25.
- **Реальные метрики (Lighthouse)** — настоящие Core Web Vitals (LCP/CLS/TBT) на главной
  вместо «оценочно»; категории получают `confidence: full`.
- **История + Diff** — каждый аудит сохраняется (`data/audits/`), два прогона сравниваются:
  «было C(72) → стало B(81)», что закрыто / появилось.
- **Страницы под логином** — cookie/заголовки (same-origin, сбрасываются на кросс-origin
  редиректе, не пишутся в историю/логи).

Глубокий режим и Lighthouse — **опциональны** (движок остаётся zero-dep, пакеты не в
`package.json`). Установка:

```bash
npm i playwright axe-core lighthouse chrome-launcher
npx playwright install chromium
```

После установки в UI активируются тогглы **«Глубокий режим»** и **«Реальные метрики
(Lighthouse)»** (иначе — неактивны с подсказкой), в CLI — флаги `--deep` / `--lighthouse`.
Egress-защита распространяется и на запросы браузера (metadata блокируется всегда). Без
установки статический путь работает как прежде (zero-dependency).

## Тесты

```bash
npm test                 # движок: node test/run.js — 106 герметичных тестов (без сети наружу)
cd web && npm test       # фронт: node --test — маппинг сообщений об ошибках по кодам
```

Покрыто: SSRF-защита (IPv4/IPv6, metadata, mapped/NAT64), декомпрессия
(gzip/deflate/raw-deflate/brotli), редиректы/таймауты/лимит распаковки/байт-бюджет обхода,
кодировки (utf-8/windows-1251), прощающий парсер на кривом HTML, обход и калибровка скоринга
(плохой → D/F, хороший → A/B, мёртвый сайт → F), недостижимый старт → код-ошибка,
классификация внешних ссылок, и весь SSE-протокол сервера (форма каждого события по контракту).

## Архитектура

```
.
├── server.js            HTTP-сервер: bind 127.0.0.1, статика web/dist + SSE-аудит
├── cli.js               запуск аудита из терминала
├── src/
│   ├── net-guard.js     egress/SSRF-защита (проверка IP + lookup-пиннинг)
│   ├── fetcher.js       запрос: редиректы, gzip/br, charset, общий тайм-аут
│   ├── html.js          прощающий токенайзер + модель страницы
│   ├── crawler.js       BFS-обход своего хоста (лимит, параллелизм, троттлинг)
│   ├── audit.js         оркестратор: обход → проверки → скоринг → отчёт
│   ├── score.js         потолки/режимы/веса, баллы и оценка
│   └── checks/          seo · tech · performance · accessibility · responsive · external
├── web/                 фронтенд (исходники) + web/dist (сборка, её отдаёт сервер)
├── test/                фикстуры + run.js
├── CONTRACT.md          контракт данных фронт ↔ движок (SSE + схема отчёта)
└── PLAN.md              полный план проекта
```

Фронт и движок общаются строго по `CONTRACT.md`: SSE-эндпоинт
`GET /api/audit/stream?url=&limit=&checkExternal=&allowLocal=` шлёт события
`meta · progress · done · error`; `done` отдаёт отчёт по схеме §3.

## Безопасность

- **Ingress:** сервер слушает только loopback (`127.0.0.1`).
- **Egress / SSRF:** фетчер ходит по произвольным URL, поэтому каждое назначение
  (стартовый URL, **каждый хоп редиректа**, каждая внешняя проверка) проверяется по
  числовому IP — блокируются loopback, link-local, RFC1918, ULA, CGNAT, IPv4-mapped
  и NAT64-формы IPv6; cloud-metadata (`169.254.169.254`) блокируется **всегда**.
  DNS-имена резолвятся один раз и коннект пинится на проверенный IP (анти-rebinding).
  Снять ограничение можно явным флагом `--allow-private` / «разрешить локальные адреса».
- **Анти-gzip-бомба:** лимит размера распакованного тела; **общий тайм-аут** запроса.

## Ограничения (и развитие, v2)

- **Статический анализ** покрывает ~80% реальных проблем и работает быстро/стабильно.
  Проверки, которым нужен рендер браузера (ошибки JS-консоли, точный контраст,
  скриншоты, полноценный Lighthouse), вынесены в опциональную «Фазу 2» на Puppeteer.
- **SPA-сайты** (контент рисуется на клиенте): движок ставит честный баннер и
  переклассифицирует находки «нет h1/alt/тонкий контент» из critical в предупреждение,
  а не сыпет ложными критичными.
- **Метрики скорости (TTFB/время)** — оценочные, с точки замера, поэтому в балл не
  входят (показываются как «оценочно»). В балл идут воспроизводимые сигналы: сжатие,
  размеры, кэш, рендер-блок и т.п.
- **Большие сайты:** обход — выборка до лимита страниц; рядом с баллом честно
  показывается «проверено N из ~M» и бейдж «выборка».
- **Кодировки не-utf8** (cp1251/koi8-r) опираются на полный ICU в стандартной сборке
  Node; на редкой сборке `small-icu` такие страницы деградируют.
